# VoCat ws_meeting_demo 评估报告

**测试日期**: 2026-05-25  
**测试环境**: macOS → clare.vinex.top/voice-api（服务端真实）  
**测试方式**: Python WebSocket 客户端，macOS `say -v Tingting` 生成中文 PCM，16kHz 16bit mono  
**样本数量**: 10 条  
**通过率**: 10/10 (100%)

---

## 1. 端到端延迟（P50 / P95）

> 计时起点：客户端发送 `{"type":"end_of_speech"}`（即 EOS 发出时刻）

| 阶段 | P50 | P95 | 平均 | 说明 |
|------|----:|----:|-----:|------|
| EOS → ASR 转录 | **1087 ms** | **2665 ms** | 1334 ms | 服务端 ASR 识别返回 |
| EOS → 首帧 LLM 文本 | **1222 ms** | **3401 ms** | 1519 ms | LLM 开始流式输出 |
| EOS → 首帧 TTS 音频 | **3293 ms** | **5131 ms** | 2926 ms | 可播放的第一包 MP3 |
| EOS → Done（全部完成）| **4824 ms** | **7880 ms** | 4367 ms | 服务端发送 done 事件 |

**关键观察**：
- 短句（TC-02/07/08）延迟显著低：EOS→首帧音频仅 1554–1721ms
- 长句（TC-01/TC-10）EOS→音频达 5131ms，主要耗时在 ASR（2458ms）+ LLM 生成
- TC-10（全国天气，3058ms 音频）done 达 7880ms，因 TTS 内容长（60 个 audio chunk）

## 2. ASR 字错率（CER）

> CER（Character Error Rate）= 编辑距离 / 参考字数，使用中文字符级计算

| ID | 场景 | 参考文本 | ASR 输出 | CER |
|----|------|---------|---------|----:|
| TC-01 | 正常长句-天气查询 | 今天北京的天气怎么样 | 今天北京的天气 | **30.0%** ⚠️ |
| TC-02 | 短句-好的 | 好的 | 好的 | **0.0%** ✅ |
| TC-03 | 长句-设置闹钟 | 帮我设置一个明天早上八点的闹钟 | 帮我设置一个明天早上8点的 | **18.8%** ⚠️ |
| TC-04 | 打断后新问题 | 停止帮我查一下明天的会议 | 停止帮我查一下明天 | **25.0%** ⚠️ |
| TC-05 | 查会议日历 | 帮我查一下明天有没有会议 | 帮我查一下明天有没有会议 | **0.0%** ✅ |
| TC-06 | 调音量指令 | 把音量调大一点 | 把音量调大一点 | **0.0%** ✅ |
| TC-07 | 查时间-短句 | 现在几点了 | 现在几点了 | **0.0%** ✅ |
| TC-08 | 讲笑话 | 给我讲个笑话 | 给我讲个笑话 | **0.0%** ✅ |
| TC-09 | 翻译指令 | 请把这句话翻译成英文 | 请把这句话翻译成英文 | **0.0%** ✅ |
| TC-10 | 全国天气-长句 | 今天全国各地的天气情况怎么样 | 今天全国各地的天气情况怎么样 | **0.0%** ✅ |

**汇总**：
- 平均 CER：**6.8%**（去掉标点后字符级）
- 完美识别（CER=0%）：7/10
- CER > 15% 的问题案例：3/10（TC-01, TC-03, TC-04）

**高 CER 根本原因分析**：

| 案例 | 问题 | 根本原因 |
|------|------|---------|
| TC-01（30%） | "怎么样" 丢失 | 句尾词被 FLUSH_FRAMES（300ms）切掉，ASR 未收到完整音频 |
| TC-03（18.8%） | "闹钟" 丢失 + 数字识别 | 同上；"八" → "8" 属数字格式化差异，可接受 |
| TC-04（25%） | "的会议" 丢失 | 同上——长句末尾词系统性被截断 |

> **结论**：`FLUSH_FRAMES=15`（300ms）对 2-3s 的中文长句不够，末尾 1-2 个词被截掉。**建议将 FLUSH_FRAMES 提高至 25（500ms）**。

## 4. VAD 相关观察（服务端视角）

本次测试使用 Python 客户端直接发送 EOS，绕过了设备端 VAD。以下为基于日志推断的 VAD 行为预测：

| 场景 | 预期 VAD 行为 | 当前 FLUSH_FRAMES=15 风险 |
|------|-------------|--------------------------|
| 短句 ≤500ms | MIN_AUDIO_FRAMES_FOR_EOS=40 抑制 | TC-02 "好的" 可能被噪声门抑制 |
| 长句 >2000ms | 正常触发 | 末尾词截断（已验证） |
| TTS 结束后立即说话 | 1000ms cooldown 保护 | cooldown 期间说话丢失 |
| 打断后首句 | s_interrupt_flag 在 receive 前清除 | 已修复（2026-05-22） |

## 5. 已知问题与建议

### 问题 P0：长句末尾词系统性截断
- **影响**：3/10 用例 CER >15%，用户指令被截断导致理解错误
- **根因**：`FLUSH_FRAMES=15`（300ms）不足以覆盖 ASR 缓冲的最后 1-2 个字
- **修复**：`host_ws.c:67` `#define FLUSH_FRAMES 25   // 25 x 20ms = 500ms`
- **验证**：重测 TC-01/03/04，CER 预期降至 <5%

### 问题 P1：EOS→首帧音频 P95=5131ms 偏高
- **影响**：用户感知响应慢（>5s 才听到声音）
- **定位方向**：需区分 ASR 延迟 vs LLM TTFT vs TTS 首包延迟
- **方法**：在服务端分别记录各阶段时间戳，或在 on_host_msg 增加 `answer_text` → `answer_audio` 差值日志
- **当前已知**：TC-01 EOS→ASR=2458ms，ASR 单独消耗了 47% 的总延迟

### 问题 P2：TC-10 done=7880ms（60 个 audio chunk）
- **影响**：长回答（全国天气、讲笑话）总交互时间可达 8-10s
- **建议**：检查 TTS 是否支持流式播放（边收边播），确认设备端 mp3_player 队列（QUEUE_DEPTH=64）不会成为瓶颈
